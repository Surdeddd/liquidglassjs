## 2. Per-repo limitations (from issues + README caveats)

**rdev/liquid-glass-react** — the market leader by downloads, effectively **abandoned demo-ware**. Open issues (none answered by maintainer): React 19-only peer dep breaks React 18 installs (#13, #25, #30); forced `translate(-50%,-50%)` center-anchoring makes normal layout placement nearly impossible (#20); shader mode renders broken stripes (#29); layers misplace with flex/Tailwind (#22, #13); button stays stretched after mouse leave (#19); frame drops on M2 Chrome (#3); an issue literally titled "unusable. made purely for demo" (#23); no true edge refraction in default modes (#11); `overLight` doesn't auto-adapt text color — accessibility complaint (#14). Its 97k monthly downloads are the clearest proof of unmet demand.

**liquid-dom** — technically the most ambitious (WebGPU renderer, SwiftUI-like layout engine, Three/R3F adapters, 5-package monorepo), but live-DOM glass requires **experimental HTML-in-Canvas behind a Chrome flag** — not shippable to end users today. No Safari/Firefox story at all.

**liquidGL** — best vanilla DX and true shared-canvas architecture, but: html2canvas snapshot means **CSS animations aren't refracted** and dynamic content must be manually registered; confirmed bug: refraction drift on long pages with multiple in-flow lenses (#8, snapshot height desync — fix PR sat unmerged for months); `position: sticky` unsupported; intermittent blank glass on first load (#4); all lenses must share one z-index; fixed elements ignored; GPU texture limits on long pages; Safari unstable when lens >50% of viewport.

**samasante/liquid-glass** — the strongest cross-browser story (live DOM in all 3 engines) but cross-browser *bending* requires the developer to **choose a mode** (wrap children / refract a copy / WebGL for media) — there's no single "just works over anything" mode, because `backdrop-filter: url()` physics don't allow it. Safari forced to 1× filter resolution; Safari video needs the WebGL path. React-only.

**ybouane/liquidglass** — the most honest limitations list in the field: glass must be a **direct child of the root**; no nested glass; root's own background invisible to the effect; injected canvas as first child; one WebGL context per instance (browser cap ~16); webfonts must be CORS-loaded before init; init is async 100–500ms; canvas/img changes need manual `markChanged()`.

**Specy/liquid-glass** — real three.js physics (actual IOR) and surprisingly strong downloads (12.4k/mo), but full three.js in the bundle, expensive init, and a **static screenshot** backdrop with manual refresh — content under the glass goes stale.

**dashersw/liquid-glass-js** — nested glass sampling is unique, but html2canvas snapshot, no npm bundle of its own (roadmap unchecked), vanilla-only, imperative class API, global mutable `window.glassControls`.

**shuding / lucasromerodb / archisvaze / nikdelvin / dpawlikowski / deepika-builds** — the SVG/CSS "Chromium-only club": all use SVG filters inside `backdrop-filter`, all silently degrade to flat blur in Safari/Firefox. deepika-builds (newest, July 2026) is notable for the cleanest 1-file API and documented sRGB gotcha, warns against elements >800px/side (O(w×h) map regen).

**PallavAg/liquid-glass-web-react** — small but technically excellent: cross-browser live-DOM refraction, quarter-symmetry map computation, Safari filter-ID cache-busting, iOS `userSpaceOnUse` workaround. Limitations: refracts **its own children only** (no arbitrary backdrop), Safari source-size cap, Safari `<video>` unfilterable.

**iyinchao/liquid-glass-studio** — visually the closest to real iOS 26 (blob merge, springs, superellipse, Fresnel, dual WebGL2/WebGPU) but it's a **studio/playground, not a library**, and it refracts its own image/video background, never your DOM. "UI content inside shape" is still on its TODO.

## 3. Technique landscape (the key technical problem: sampling the backdrop)

There are exactly five known ways to get refraction over web content, and every library is a point in this trade-off space:

1. **`backdrop-filter: url(#svgFilter)`** — true live backdrop, zero copies. **Chromium-only**; Safari/Firefox parse the property and silently drop the SVG part → flat blur. Not in any spec. WebKit bug 245510 open for years; Mozilla Connect idea open; **W3C standardization was only just requested in w3c/svgwg#1142 (filed June 25, 2026)** with zero vendor response so far. Chromium also runs SVG filters on GPU; Safari re-rasterizes in software each frame (why samasante forces 1× there).
2. **`filter: url()` on the content itself** (Aave technique — the write-up that samasante, PallavAg, deepika-builds and Outpace all credit). `feDisplacementMap` via plain `filter` works in **all browsers**, but you must own what you refract: wrap the content as children, or maintain a counter-positioned live **copy** of the backdrop (Outpace's approach: copy tracks backdrop frame-for-frame). Displacement maps computed from SDF/Snell's-law ray tracing on canvas, R/G = XY displacement, B = baked specular, quarter-symmetry optimization.
3. **WebGL + DOM screenshot** (html2canvas / html-to-image): liquidGL, dashersw, Specy, ybouane, JKerbin. Works everywhere, best optical shaders, but the content under glass is a **frozen raster** — stale text, no CSS animations, CORS/webfont landmines, expensive re-captures. ybouane's dirty-region tracking is the state of the art for keeping snapshots fresh.
4. **WebGL/WebGPU with owned background** (studio, ektogamat, Shadertoy ports): perfect optics, useless over arbitrary DOM.
5. **WebGPU + HTML-in-Canvas** (`<canvas layoutsubtree>` + `copyElementImageToTexture()`): live DOM into a GPU texture with text selection and hit-testing preserved — the endgame architecture. Today: Chrome Canary/Brave behind `chrome://flags/#canvas-draw-element` (WICG spec, Chrome origin trial). liquid-dom and jeantimex are the only two implementations.

**Shared-single-canvas-overlay architecture:** partially exists — liquidGL (one canvas, all lenses, but same-z-index constraint + static snapshot), ybouane (per-root compositing canvas with glass-sees-glass layering), samasante (one WebGL renderer, N lenses — but only over a single video/canvas surface), liquid-dom (true WebGPU compositor + scene graph — flag-gated). **Nobody has a production shared compositor over live DOM.**

## 4. Gaps nobody covers (differentiation surface)

1. **Automatic tiered rendering with one API.** No library auto-detects capability and routes: `backdrop-filter:url()` (Chromium) → `filter:url()` copy/wrap mode (Safari/FF) → WebGL snapshot → plain blur. samasante has the pieces but makes the developer pick modes manually; everyone else picks one lane. A capability-probing engine with a single declarative surface would instantly beat all of them.
2. **True multi-framework from one core.** The 97k-download leader is React-19-only. There is no lib shipping vanilla + React 18/19 + Vue + Svelte + web-component adapters over one framework-agnostic engine (PallavAg's `LiquidGlassEngine` split shows the pattern; nobody executes it fully).
3. **Blob merge / shape morphing over live DOM.** The iOS 26 signature move — two glass elements melting together (tab bar morph) — exists only in own-background demos (studio's SDF smooth-min, jeantimex's merge strategy). No production library does metaball merging of lenses over page content.
4. **Physics as a first-class system.** Elasticity (rdev), wobble utils (samasante), spring press (jeantimex demo) are fragments. Nobody ships a coherent spring/inertia/gel system: press-squash, drag-stretch, release-settle, morph-between-shapes, with reduced-motion compliance.
5. **Adaptive contrast / luminance sampling.** Apple's glass samples the backdrop and flips tint + label color for legibility. On the web this is a manual `overLight` boolean (rdev, complained about in issue #14). Auto-sampling backdrop luminance (cheap canvas readback or ancestor color heuristics) for a11y-safe auto-theming: **zero implementations**.
6. **Accessibility, period.** No library respects `prefers-reduced-motion` or `prefers-reduced-transparency`, none mention WCAG contrast under glass. Free differentiation with actual moral high ground.
7. **Glass-on-glass nesting over live DOM.** Only snapshot libs (ybouane layered, dashersw parent-sampling) attempt it; no live-DOM lib supports a glass toolbar containing glass buttons that refract through both layers.
8. **Scroll/perf hygiene.** Sticky/fixed positioning breaks liquidGL; long pages break texture limits; no lib does viewport-culled lens rendering, device-tier auto-degradation, or an FPS budget governor (only ybouane exposes `.fps` at all).
9. **WebGPU progressive enhancement.** liquid-dom bet everything on the flag-gated future; snapshot libs are stuck in the past. Nobody ships: HTML-in-Canvas when available → SVG/WebGL fallback today, with identical API — i.e., a lib whose fidelity silently improves as browsers ship the spec.
10. **SSR-safety + strict TS + design tokens.** Only PallavAg and samasante mention SSR. No theming/token story anywhere (einui is tokens-only with fake glass). No headless + styled two-tier offering except samasante's early attempt.
11. **Tooling loop.** Studio-quality parameter playground (iyinchao) and production libraries are disjoint universes; a lib with a built-in `/playground` that exports typed config would own the workflow. (imggion's generator exports CSS only.)

**Build-vs-beat verdict:** the field is fragmented across four incompatible rendering strategies, the download leader is unmaintained and React-19-locked, the only cross-browser live-DOM libs (samasante, PallavAg) are single-framework and month-old-small, and the WebGPU endgame (liquid-dom) is flag-gated. A framework-agnostic core with capability-tiered rendering (gap 1) + physics/morphing (gaps 3-4) + adaptive contrast & a11y (gaps 5-6) has no direct competitor — every existing lib would be a strict subset.
